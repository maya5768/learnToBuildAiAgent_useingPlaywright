מה זה MCP? (Model Context Protocol)

הסבר בגובה העיניים, בלי מונחים מסובכים

💡
דמיינו תרגום לחיים: אתה עובד בחברה ויש לך עוזר (ה-AI). העוזר חכם מאוד, אבל הוא יושב בחדר אחר ולא יכול לגשת לשום מחשב. MCP הוא כמו טלפון שמחבר את העוזר לכל המערכות — הוא יכול עכשיו להסתכל על Azure DevOps, לפתוח Figma, להריץ בדיקות, ולכתוב Excel — הכל בלי שאתה צריך לדלג בין כלים.
📋 הגדרה פשוטה

MCP הוא פרוטוקול (=סט של חוקים) שמאפשר לכלי AI כמו Claude לדבר עם תוכנות חיצוניות. במקום שה-AI ידע רק מה הוא למד באימון — הוא יכול לשאול בזמן אמת: "מה יש בכרטיס הזה ב-Azure DevOps?" או "מה העיצוב ב-Figma?"

🤔 למה זה חשוב לנו?

בלי MCP: אתה פותח Azure DevOps, מעתיק ידנית את הדרישות, פותח Figma, מסתכל על העיצוב, חוזר, כותב Test Plan ב-Excel — שעה של עבודה.

עם MCP: אתה כותב פרומפט אחד ל-AI, והוא עושה את כל זה אוטומטית.

🏗️
אנלוגיית קוד: MCP הוא כמו USB Port של המחשב — הוא תקן אחיד שמאפשר לכל מכשיר (Azure DevOps, Figma, Playwright) להתחבר לאותו AI. בלי תקן כזה, כל כלי היה צריך שפה משלו.
📅 עובדה: MCP פותח על ידי Anthropic (יוצרי Claude) ב-2024 ומהר מאוד הפך לסטנדרט בתעשייה. כיום יש מעל 10,000 MCP Servers.

ארבעת הכלים בשרשרת

כל כלי עושה דבר אחד שהוא הכי טוב בו

📋
Azure DevOps
מערכת ניהול פרויקטים של Microsoft. כאן מנהלים משימות (Work Items), כרטיסים, Sprint, ובדיקות. מקבילה ל-Jira.
🎨
Figma
כלי עיצוב בדפדפן. מעצבים מציירים בו מסכים, פלואים וקומפוננטות. ממנו מקבלים את ה"איך הדבר צריך להיראות".
🎭
Playwright
כלי של Microsoft לאוטומציית דפדפן. מריץ בדיקות UI אוטומטיות — כמו רובוט שלוחץ על הכפתורים במקומך.
📊
Excel (.xlsx)
הפלט הסופי — קובץ Test Plan מסודר עם עמודות: ID, כותרת, שלבי בדיקה, תוצאה מצופה. ניתן לייבא חזרה ל-ADO.
🤖

ה-AI — המנצח על התזמורת

Claude (או GitHub Copilot) מחבר את כולם

ה-AI לא מחליף את הכלים — הוא מתאם ביניהם. הוא קורא מ-ADO, מפרש את Figma, ומייצר תוכן Test Plan, ואז כותב Excel. בלי MCP הוא לא יכול לגשת לשום דבר מאלה.

כלי MCP Server מה ה-AI יכול לעשות איתו
Azure DevOps @azure-devops/mcp קרוא Work Items, צור Test Plans, עדכן כרטיסים
Figma Figma Dev Mode MCP קרוא קומפוננטות, מסכים, flows, spacing tokens
Playwright @playwright/mcp הרץ בדיקות דפדפן, צלם screenshots, ולידציה
Excel קוד Python/JS (אין MCP) צור קובץ .xlsx עם Test Cases מסודרים
❓ מה תפקיד Playwright בשרשרת?
⛓️

שרשרת הפעולות — צעד אחרי צעד

מכרטיס ADO ועד קובץ Excel בסוף

1
📋 קריאת Work Item מ-Azure DevOps ADO MCP

ה-AI מתחבר דרך MCP לאורגניזציה שלך ב-ADO ומושך את כרטיס המשימה. הכרטיס מכיל: תיאור הפיצ'ר, Acceptance Criteria, Priority, ו-Story Points.

דוגמה: Work Item #1234 — "כפתור Login חדש עם אימות Google"

2
🎨 קריאת עיצוב מ-Figma Figma MCP

ה-AI ניגש לקובץ Figma הרלוונטי ומחלץ: מסכים, flow של המשתמש, מצבים שונים (empty state, error state, success) ו-interactions.

דוגמה: מוצא 3 מסכים — דף Login, דף Error, דף Success עם כפתור Google

3
🤖 יצירת Test Cases על ידי ה-AI AI (Claude / Copilot)

ה-AI משלב את המידע מ-ADO ומ-Figma ויוצר Test Cases מפורטים. כל Test Case כולל: שם הבדיקה, תנאי התחלה, שלבי הבדיקה, תוצאה מצופה.

דוגמה: "TC01 — לחיצה על כפתור Google → מופיע popup → אישור → redirect לדף הבית"

4
🎭 ולידציה עם Playwright (אופציונלי) Playwright MCP

שלב אופציונלי — Playwright פותח דפדפן ובודק שהאפליקציה באמת עובדת. הוא יכול לצלם screenshots ולהשוות לעיצוב מ-Figma. ה-AI אחר כך מתאים את ה-Test Cases לפי מה שמצא.

5
📊 ייצוא ל-Excel (.xlsx) פלט סופי

ה-AI מייצר קובץ Excel מסודר עם כל ה-Test Cases. העמודות תואמות לפורמט שניתן לייבא חזרה ל-Azure Test Plans.

עמודות: TC ID | כותרת | תיאור | Pre-conditions | שלבים | תוצאה מצופה | Priority

⚡
כמה זמן חוסכים? תהליך ידני לוקח 2-4 שעות. עם ה-workflow הזה — כ-10-15 דקות. ה-AI מייצר Draft, ואתה רק מוודא ועורך.
⚙️

ההגדרה הטכנית — איך מחברים הכל

קוד אמיתי, מוסבר שורה-שורה

📁 שלב 1 — יצירת קובץ mcp.json

זה הקובץ שאומר ל-AI אילו כלים הוא יכול להשתמש בהם. יוצרים אותו בתיקיית .vscode בפרויקט.

📄 .vscode/mcp.json
{
  "inputs": [
    {
      "id": "ado_org",          // שם הארגון ב-Azure DevOps
      "type": "promptString",    // ישאל אותך כשתפעיל
      "description": "שם הארגון ב-ADO (לדוגמה: contoso)"
    }
  ],
  "servers": {

    // ── שרת Azure DevOps ──
    "ado": {
      "type": "stdio",
      "command": "npx",
      "args": [
        "-y",
        "@azure-devops/mcp",  // חבילת ה-MCP
        "${input:ado_org}",   // שם הארגון שהזנת
        "-d",
        "core",              // domain: מידע בסיסי
        "work-items",        // domain: כרטיסי עבודה
        "test-plans"         // domain: תכניות בדיקה
      ]
    },

    // ── שרת Playwright ──
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": [
        "@playwright/mcp@latest"  // Playwright MCP
      ]
    }

    // Figma: מופעל מתוך אפליקציית Figma Desktop
    // Settings → Enable Dev Mode MCP Server
  }
}
💡 מה זה npx? זה כלי של Node.js שמוריד ומריץ חבילות JavaScript בלי שצריך להתקין אותן מראש. כמו "הורד והפעל עכשיו".
🔑 שלב 2 — אימות (Authentication)

Azure DevOps דורש Personal Access Token (PAT) — סוג של סיסמה מיוחדת. יוצרים אותה ב-ADO תחת User Settings → Personal Access Tokens. את הטוקן שמים במשתנה סביבה:

💻 Terminal (Windows PowerShell)
# מגדירים את הטוקן כמשתנה סביבה
$env:AZURE_DEVOPS_EXT_PAT = "הטוקן-שלך-כאן"

# או ב-Mac/Linux:
export AZURE_DEVOPS_EXT_PAT="הטוקן-שלך-כאן"
🐍 שלב 3 — קוד Python ליצירת Excel

האחרון בשרשרת — מקבלים את ה-Test Cases שה-AI יצר וכותבים לקובץ Excel. משתמשים בספריית openpyxl:

🐍 generate_test_plan.py
import openpyxl
from openpyxl.styles import Font, PatternFill, Alignment

# יצירת קובץ Excel חדש
wb = openpyxl.Workbook()
ws = wb.active
ws.title = "Test Plan"

# הגדרת כותרות העמודות
headers = [
    "TC ID",        # מספר הבדיקה
    "כותרת",        # שם הבדיקה
    "Pre-condition", # תנאי התחלה
    "שלבים",        # מה עושים
    "תוצאה מצופה",  # מה אמור לקרות
    "Priority"       # דחיפות
]

# כתיבת הכותרות עם עיצוב
for col, header in enumerate(headers, 1):
    cell = ws.cell(row=1, column=col, value=header)
    cell.font = Font(bold=True, color="FFFFFF")
    cell.fill = PatternFill("solid", fgColor="0078D4")  # כחול ADO

# דוגמה: הוספת Test Case
test_cases = [
    ["TC001", "כניסה עם Google", "משתמש לא מחובר",
     "1. לחץ Login\n2. בחר Google\n3. אשר הרשאות",
     "redirect לדף הבית עם שם המשתמש", "High"],
]

for row_idx, tc in enumerate(test_cases, 2):
    for col_idx, value in enumerate(tc, 1):
        ws.cell(row=row_idx, column=col_idx, value=value)

wb.save("test_plan.xlsx")
print("✅ קובץ Excel נוצר בהצלחה!")
📦
להתקין את הדרוש: כדי להריץ את הקוד, צריך Python ואת הספרייה: pip install openpyxl

פרומפטים — מה כותבים ל-AI

פרומפטים מוכנים לשימוש לכל שלב בתהליך

🔵 שלב 1 — שליפת Work Item מ-ADO
🤖 PROMPT לבדיקה
קרא את Work Item מספר 1234 מ-Azure DevOps ותחלץ ממנו:
1. כותרת המשימה
2. תיאור מפורט
3. Acceptance Criteria (קריטריוני קבלה)
4. Priority
סכם בפורמט נקי שנוכל להשתמש בו ליצירת Test Plan.
🟣 שלב 2 — ניתוח Figma
🤖 PROMPT לעיצוב
היכנס לקובץ Figma עם ID: [FIGMA_FILE_ID] ומצא את ה-Frame בשם "Login Flow".
חלץ:
1. רשימת המסכים / מצבים (states) שקיימים
2. User Flow — מה קורה כשלוחצים על כל כפתור
3. הודעות שגיאה שמוצגות
4. מצבי קצה (edge cases) שרואים בעיצוב
🟡 שלב 3 — יצירת Test Cases
🤖 PROMPT ליצירה
בהתבסס על:
- Work Item: [כותרת + Acceptance Criteria שחולצו בשלב 1]
- עיצוב Figma: [רשימת מסכים ו-flows מהשלב 2]

צור Test Plan מקיף הכולל:
- בדיקות Happy Path (המסלול הרגיל שעובד)
- בדיקות Negative (מה קורה כשמשהו לא תקין)
- בדיקות Edge Cases (מקרי קצה)

פורמט כל Test Case: TC_ID | כותרת | Pre-condition | שלבים | תוצאה מצופה | Priority
🟢 שלב 4 — ייצוא ל-Excel
🤖 PROMPT לייצוא
קח את ה-Test Cases שיצרת וכתוב קוד Python שמייצר קובץ Excel (test_plan.xlsx) עם:
- שורת כותרות מעוצבת בצבע כחול
- כל Test Case בשורה נפרדת
- עמודות: TC_ID, Title, Pre-conditions, Steps, Expected Result, Priority
- הפורמט תואם לייבוא ל-Azure DevOps Test Plans
🎭 שלב אופציונלי — Playwright

אפשר להוסיף ולידציה אוטומטית לפני יצירת ה-Excel:

🤖 PROMPT לולידציה
השתמש ב-Playwright כדי לפתוח את [URL האפליקציה] ולבדוק:
1. האם כפתור "Login עם Google" קיים בדף?
2. צלם screenshot של הדף הנוכחי
3. לחץ על הכפתור — האם מופיע popup אימות?
עדכן את ה-Test Cases בהתאם למה שמצאת.

מילון מונחים

כל המונחים הטכניים מוסברים בפשטות

MCP — Model Context Protocol
פרוטוקול תקשורת בין AI לכלים חיצוניים. כמו שפה משותפת שמאפשרת ל-AI לדבר עם Azure DevOps, Figma ועוד.
MCP Server
תוכנה קטנה שרצה על המחשב שלך ומתווכת בין ה-AI לשירות ספציפי (לדוגמה: @azure-devops/mcp הוא MCP Server ל-Azure DevOps).
Work Item
כרטיס משימה ב-Azure DevOps. יכול להיות User Story, Bug, Task, Feature. מכיל תיאור, דרישות ו-Acceptance Criteria.
Acceptance Criteria
רשימת תנאים שצריכים להתקיים כדי שהפיצ'ר ייחשב "גמור". לדוגמה: "כשלוחצים Login עם Google, המשתמש מגיע לדף הבית תוך 3 שניות".
Test Plan
מסמך שמפרט אילו בדיקות יש לבצע על פיצ'ר. מכיל Test Cases — כל אחד עם שם, שלבים ותוצאה מצופה.
Test Case
בדיקה ספציפית אחת. לדוגמה: "TC001 — בדיקת כניסה עם Google — לחץ כפתור → בחר חשבון → ציפיה: redirect לדף הבית".
Happy Path
המסלול הרגיל שעובד כמצופה. הבדיקה שמכסה מה קורה כשהמשתמש עושה הכל נכון.
Edge Case
מקרה קיצוני שקורה לעיתים נדירות. לדוגמה: מה קורה אם המשתמש מזין סיסמה של 1000 תווים?
PAT — Personal Access Token
סיסמה מיוחדת שנוצרת עבור API. בניגוד לסיסמה רגילה, ניתן להגביל אותה להרשאות ספציפיות ולתקופת זמן.
stdio
שיטת תקשורת בין תוכנות דרך "קלט/פלט סטנדרטי". ה-MCP Server מקבל פקודות ומחזיר תשובות בצינור תקשורת זה.
Domain (ב-ADO MCP)
קבוצת כלים קשורים. לדוגמה, domain "work-items" כולל כלים לקריאה/כתיבה של כרטיסים. מגדירים אילו domains לטעון כדי לא לטעון יותר מדי.
npx
כלי ב-Node.js שמוריד ומריץ חבילות JavaScript מהאינטרנט ללא התקנה מראש. כמו "הורד והפעל עכשיו".
